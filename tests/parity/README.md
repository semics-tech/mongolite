# Parity tests

These tests run the same operation against **mongolite-ts** and a **real MongoDB instance**
(via [`mongodb-memory-server`](https://github.com/typegoose/mongodb-memory-server)) and assert
the two return identical results. Unlike the rest of the test suite, correctness here isn't
defined by a hand-written expected value â€” it's defined by what real MongoDB actually does. That
matters because a hand-written assertion can only ever check that the code matches the author's
belief about MongoDB's behavior; if that belief is wrong, the test just encodes the same mistake
and passes forever. This is exactly how `find().project({})` shipped returning only `_id`
instead of the full document.

## Running

```sh
npm run test:parity
```

The first run downloads a real `mongod` binary (cached afterwards in `~/.cache/mongodb-binaries`).
This requires network access; if you're offline, the download will fail â€” CI runs this same
suite in `.github/workflows/parity.yml` with the binary cached and `MONGOMS_VERSION` pinned for
determinism.

This is a separate script from `npm test` â€” parity tests are never included in `npm test`,
`npm run test:coverage`, or `npm run test:coverage:check` (see `package.json`), so they don't
affect coverage thresholds and won't unexpectedly run for anyone working offline.

## Adding a scenario

Scenarios are plain data, not test code â€” add an entry to the relevant array in
`tests/parity/scenarios/*.scenarios.ts` (or create a new file for a new feature area):

```ts
{
  description: 'what this scenario checks, in plain English',
  seedDocs: [ /* documents with explicit, deterministic _id values */ ],
  operation: { kind: 'find', filter: { ... }, projection: { ... } },
}
```

`operation.kind` can be `find`, `findOne`, `updateOne`, `updateMany`, or `aggregate`. For
`updateOne`/`updateMany`, the harness re-reads the whole collection after the write and compares
that (`verifyVia: 'refetchAll'`, the default for writes) rather than diffing the update
acknowledgement, since ack shapes aren't meaningfully comparable between the two drivers.

Use explicit `_id` values in `seedDocs` â€” this sidesteps the need to normalize MongoDB's BSON
`ObjectId` type against mongolite's plain string `_id`, since only autogenerated `_id`s need that
normalization (`tests/parity/harness/normalize.ts`).

If you're deliberately documenting a known, intentional divergence from real MongoDB (rather than
a bug to fix), add `knownDivergence: { reason: '...' }` to the scenario â€” it still runs, but
asserts the two results are *different*, so if the divergence quietly disappears or MongoDB's own
behavior changes, the scenario fails instead of silently going stale.

## How it works

- `harness/backend.ts` defines a small common interface (`seed`, `run`, `dropCollection`,
  `close`) implemented once for mongolite-ts (`harness/mongoliteBackend.ts`) and once for the
  real driver (`harness/mongoBackend.ts`).
- `harness/setup.ts` lazily starts a `MongoMemoryServer` the first time any parity test file
  needs it.
- `harness/normalize.ts` normalizes BSON-specific types (`ObjectId` â†’ hex string, `Date` â†’ ISO
  string) before comparing, and sorts result sets by `_id` when the operation has no explicit
  `sort` â€” neither SQLite's table-scan order nor MongoDB's natural order is a contract either
  engine promises absent one.
- `harness/parityRunner.ts` ties it together: for each scenario, seed both backends into a
  freshly-dropped, uniquely-named collection, run the operation on both, normalize, and
  `assert.deepStrictEqual`.
